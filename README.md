### Разделение по критичности

- **Критические:** гонка выбора оператора; разрыв между статусом в БД и фактической доставкой в телефонию при ретраях; риск неконсистентного состояния оператора/звонка при ошибках после частичного сохранения.
- **Важные:** отсутствие транзакционности и явной модели этапов; идемпотентность к внешней системе; поведение при «нет операторов»; индексы/порядок выбора; устойчивость HTTP-интеграции.
- **Было бы хорошо:** расширенное логирование и метрики; fairness; документация контрактов статусов; нагрузочные профили под пик.

---

### Какие тесты добавить первыми

1. **Конкурентный тест на выбор оператора** (несколько «воркеров»/транзакций в тесте): при N параллельных джобах не должно получиться двух звонков на одного оператора (или должна соблюдаться явно выбранная политика шардирования).
2. **Сценарий «телефония упала после assign в БД»:** при повторе джобы событие в телефонию должно быть отправлено ровно один раз (или осознанно идемпотентно N раз по контракту).
3. **Нет доступных операторов:** корректная задержка/повтор, отсутствие «busy loop», метрика/лог.
4. **Звонок уже обработан / удалён:** ранний выход без побочных эффектов.
5. **Контрактные тесты на `TelephonyClient`** с имитацией таймаутов и 5xx (по возможности).

### Что не стал бы делать «прямо сейчас»

- Введение распределённых транзакций (2PC) между БД и телефонией — обычно избыточно; лучше сага/outbox.
  **2PC** предполагает, что внешний HTTP‑сервис участвует в распределённом коммите наравне с БД — на практике этого нет или это даёт высокую латентность, хрупкость при сбоях и редко сочетается с таймаутами и масштабированием очередей.
  **Сага** и **outbox** сохраняют согласованность поэтапно: изменения коммитятся в БД отдельно от доставки в телефонию, ретраи и компенсирующие действия описываются явно — без попытки «одной атомарной транзакции» через сеть до чужого API.
- Сложный самописный шардированный планировщик операторов до появления данных о **fairness** и **SLA**.
  **Fairness** здесь — насколько равномерно и предсказуемо входящие звонки распределяются между доступными операторами (без «голодания» части линии и без монополизации очереди одним оператором из‑за сортировок, `NULL` в полях вроде `last_call_at` и гонок при параллельных воркерах).
  **SLA** (service level agreement) — договорённые или продуктовые целевые показатели: например, сколько секунд допустимо ждать назначения при пике, какой процент звонков должен попасть к оператору в срок, что считается отказом. Без метрик по реальным задержкам и отказам непонятно, нужен ли вообще шардированный планировщик или достаточно атомарного выбора оператора и лимитов на интеграции.
- Массовое усложнение схемы статусов без наблюдаемости (логи/метрики/трейсы) — сначала сделать поведение при ошибках предсказуемым и измеримым.

### Предположения (где поведение не описано явно)

- **Очередь Redis (Laravel):** джобы не теряются при рестарте без `failed_jobs`/DLQ; нет дублей постановки; `retry_after` больше максимального времени блокировок — *не гарантировано*.
- **Телефония:** поддерживает идемпотентные повторы или нет; есть ли callback подтверждения назначения — *неизвестно*.
- **Legacy:** уникальность `(phone)` у клиента, семантика `last_call_at`, можно ли менять `operator.available` из других мест — *не описано*.
- **Инфраструктура:** один Redis/одна БД; лимиты connection pool; региональная задержка до телефонии — *типичные production допущения*.

**Риски неопределённости:** неверный выбор между «at-least-once» и «exactly-once» на границе HTTP; скрытые дубли событий; невозможность воспроизвести баг без логов телефонии; неожиданный порядок при `NULL` в сортировке; внезапные deadlock-и при добавлении жёстких блокировок без настройки таймаутов.

### Масштабирование (рост нагрузки ×10–×50)

#### Ожидаемые bottleneck-и текущей реализации

- **Строка «найти свободного оператора»** — горячая точка: блокировки, длинные очереди ожидания, рост p99 latency маршрутизации.
- **Запись в `calls` и обновление `operators`** — конкуренция за строки и индексы; рост транзакционного конфликта.
- **Синхронный HTTP в воркере** — исчерпание concurrency воркеров, исходящий rate limit телефонии, рост очереди Redis.
- **Redis как брокер** — память под длинные хвосты очереди, latency при большом RPS dispatch, единая точка отказа без кластера.
- **Логирование** — объём и стоимость хранения, backpressure на агент сбора логов.

#### Простое увеличение числа workers

- **Помогает:** когда узкое место — CPU/ожидание I/O *вне* БД и *вне* выбора одного и того же ресурса; снижает длину очереди при умеренной нагрузке.
- **Перестаёт помогать:** при насыщении БД (CPU, IOPS, locks), при лимите телефонии (429/5xx/connection limit), при исчерпании пула соединений к Redis/БД, при гонках — больше воркеров усугубляет contention и дедлоки.

#### Лимиты по компонентам


| Компонент          | Типичные лимиты                                                                                                                     |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------- |
| **Redis**          | память, maxclients, single-thread bottleneck на узле, репликация и failover; для очередей — длина очереди и время ожидания воркера. |
| **БД**             | connections (`max_connections`), lock wait timeout, индексное покрытие hot path, репликационный лаг если читаем со слейва.          |
| **HTTP телефония** | rate limit, таймауты, keep-alive лимиты, стоимость ретраев, «thundering herd» при массовых ошибках.                                 |
| **Логи**           | лимиты ingestion, диск, стоимость, влияние на latency при синхронном логировании.                                                   |


#### План развития (высокий уровень)

1. **Наблюдаемость:** метрики (очередь, время маршрутизации, доля «нет операторов», успех телефонии), трейсинг сквозь джобу и HTTP.
2. **Корректность под параллелизмом:** атомарный выбор оператора; чёткая state machine для звонка; outbox или отдельная джоба на телефонию с идемпотентностью.
3. **Декуплинг телефонии:** очередь исходящих событий с ограниченным concurrency и backoff; согласование с контрактом телефонии.
  **Декуплинг** — развести два этапа: фиксация назначения в своей БД и доставку уведомления во внешнюю систему, чтобы сбои/латентность HTTP не блокировали основной пайплайн и ретраи касались только недоставленного вызова, а не всей маршрутизации.
   **Concurrency** — сколько отправок во внешнюю телефонию выполняется **одновременно** (параллельные воркеры, процессы, лимит «не больше N одновременных запросов» к API), чтобы не упереться в rate limit или исчерпать соединения.
   **Backoff** — паузы с нарастающей задержкой (часто с jitter) между повторами после ошибки, чтобы не забивать телефонию шквалом одинаковых запросов и дать системе восстановиться.
4. **Горизонтальное масштабирование данных:** партиционирование/шарды по региону или очереди звонков; read replicas осознанно (если допустим eventual consistency для части чтений).
5. **Redis/очередь:** кластер/отдельный брокер при росте; политики `retry_after`, dead-letter, приоритетные очереди для SLA.
6. **БД:** индексы под hot queries, пулы соединений с лимитом на процесс, возможно PgBouncer/MySQL proxy — в зависимости от СУБД.
7. **Операционные процедуры:** runbook на «залипших» операторах и необработанных звонках, канареечные выкладки.
  **Runbook** — пошаговая инструкция для дежурных/эксплуатации: типичные симптомы («оператор недоступен, но в БД занят»), как диагностировать (какие метрики, логи, запросы в БД безопасно делать), какие действия разрешены для восстановления и в каком порядке, когда эскалировать. Цель — снизить время реакции и риск ошибок «нажми кнопку в проде».
